## 示例 1：设备详情卡片（kv_grid + tabular + kv_grid）

**对应文件**：[`example_1_device_detail.yaml`](./example_1_device_detail.yaml)  
**底层网格**：8 列

### 数据填充逻辑

```
区段一（基本信息）
  → 执行 SQL，返回 1 行，fields[].col 按列名取值
  → 6 对 KV，cols_per_row=2，每行排 2 对
  → 最终 3 行数据行

区段二（关键指标）
  → 执行 SQL，返回 N 行，每行 → 表格一行
  → columns[].field 按列名取值填入各格

区段三（综合结论）
  → 执行 SQL，返回 1 行，fields[].col 按列名取值
  → 2 对 KV，cols_per_row=1，每行全宽
```

## 示例 2：多设备横向对比矩阵（tabular 动态列）

**对应文件**：[`example_2_comparison.yaml`](./example_2_comparison.yaml)  
**底层网格**：逻辑列数 = 1 + 设备数（由 SQL PIVOT 列决定）

### 数据填充逻辑

```
区段一（汇总元信息）
  → 参数直填 {time_range}、{device_count}

区段二（对比主表）
  → SQL PIVOT 查询：列名 = 设备ID，行 = 指标
  → headers 中 ~dynamic~ 列自动展开为实际设备 ID 列
  → columns 中 ~dynamic~ 列按 SQL 结果列名自动映射
  → N 台设备 = N 列，模板不写死
```

## 示例 3：动态属性面板（kv_grid key_col/value_col）

**对应文件**：[`example_3_dynamic_kv.yaml`](./example_3_dynamic_kv.yaml)  
**底层网格**：8 列

### 数据填充逻辑

```
区段一（设备标识）
  → 参数直填：fields[].value 含 {$device}、{time_range}

区段二（设备档案）
  → SQL 返回多行：每行 (attr_name, attr_value)
  → key_col="attr_name" → KV 的 key
  → value_col="attr_value" → KV 的 value
  → 行数由数据库决定，模板不预设字段数
  → 按 cols_per_row=2 自动排入多行

区段三（近期运行记录）
  → SQL 返回多行，按 columns[].field 映射列
```

## 三种填充方式对比

| **方式** | **配置字段** | **适用场景** |
|---|---|---|
| **参数直填** | `fields[].value: "{param}"` | 字段固定、值来自用户输入/全局参数 |
| **SQL 单行，字段固定** | `fields[].col` + `source.query` | 字段固定、值来自数据库（一条记录） |
| **SQL 多行，动态 KV** | `source.key_col` + `source.value_col` | 字段名也来自数据库，数量不确定 |
| **SQL 多行，tabular** | `columns[].field` + `source.query` | 明细列表，行数不确定 |
| **SQL PIVOT，动态列** | `~dynamic~` 标记 + PIVOT 查询 | 列数（如设备数）运行时才确定 |
| **AI 属性/总结** | `source.kind: ai_synthesis` | 需要引用多源上下文或知识库进行专家级总结 |

## 示例 4：带 AI 专家建议的评估表（AI 总结型）

**对应文件**：[`example_4_ai_summary.yaml`](./example_4_ai_summary.yaml)  
**设计逻辑**：RAG（检索增强生成）模式

### 填充逻辑与知识检索输入

1.  **准备上下文**:
    *   `refs`: 实时读取前一个区段「监控指标快照」的渲染结果。
    *   `queries`: 执行补充的「故障历史」SQL 查询。
2.  **构造知识检索 (Input)**:
    *   **Subject**: `{device_model}` (如：KD-PUMP-204A)
    *   **Symptoms**: 动态筛选 `realtime_metrics` 中状态为「异常」的测点作为征兆。
    *   **Objective**: 固定描述为「获取行业诊断建议」。
3.  **合成生成**: 将以上背景信息注入 `prompt` 调用 LLM。

### 渲染效果预览

```
+════════════════════════════════════════ 监控指标快照 ═════════════════════════════════════════+
│      测点        │     数值     │ 阈值 │  状态  │
│    振动幅度      │   7.8 mm/s   │ 7.5  │  异常  │
│    轴承温度      │    82.1 ℃    │ 90.0 │  正常  │
+═══════════════════════════════════ AI 专家评估与检修建议 ════════════════════════════════════+
│    评估建议    │  [AI 生成内容]：当前设备振动值(7.8)已超限。结合知识库分析，此类       │
│                │  水泵在振动超限时核心风险点在于转子不平衡。建议步骤：1. 检查基座      │
│                │  螺栓... 2. 检查轴承状态...                                         │
```
