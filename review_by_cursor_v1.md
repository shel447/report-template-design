## 设计整体评价

- **整体结构清晰**：从模板分类 → 参数系统 → 章节结构 → 内容节点类型 → 占位符约定 → Schema & 示例，层次合理，既有概念说明也有落到实现层面的约束（schema + 示例模板），便于后续实现和扩展。
- **关注扩展性**：通过 `type`/`scene` 的二级分类、参数系统的可扩展属性、`source` 与 `presentation` 的解耦，以及 `foreach` 机制，为未来新增报告类型、场景和内容形态预留了空间。
- **数据与呈现解耦良好**：将内容节点拆成数据来源（`source.kind`）与呈现形式（`presentation.type`），再配合组合矩阵约束，有助于避免「一个 content 里什么都干」的实现膨胀。
- **占位符约定统一**：`{param_id}`、`{$varname}`、`{$value}` 语法简单直观，区分了全局参数、循环变量以及 SQL 单值返回。

总体来看，文档非常适合作为「模板 schema 设计 + 内容模型设计」的第一版基础，已经具备较好的工程可落地性。

## 模糊点与潜在风险

- **模板选择与识别流程未在本文中约束**  
  当前仅描述了「匹配报告类型 & 场景」，但没有在 schema 层面描述「模板元信息」结构，例如：
  - 模板如何被唯一标识（`template_id` / `version` 等）？
  - 一个 `type` + `scene` 下如果有多个版本或子模板，如何选取（优先级、启停标记等）？
  建议在文档中明确是否由其他模块负责（如「模板管理系统」），抑或在 schema 中补充模板级元信息。

- **参数收集与校验规则略显粗粒度**  
  当前参数属性中有 `required` 标记，但缺少：
  - 默认值（`default`）及其语义（静态 / 动态计算）
  - 校验规则（例如正则、数值范围、长度限制）  
  这些在真正做「对话收参」和「表单回填」时会很关键，建议后续扩展为可选字段，以免逻辑分散到实现代码中而丢失统一规范。

- **`input_type` 与 `source` 的职责边界可再澄清**  
  - `dynamic` 类型目前通过 `source` 字符串说明选项来源，但未定义返回格式（例如：`[{value, label}]` 还是任意结构？）。
  - 一些场景中，可能希望同一个参数既能自由输入又能从枚举中选（例如「其他」），当前枚举模型下需要约定实现策略（如通过附加 `allow_free_text`）。
  可以在文档中用 1–2 个典型示例补充说明，降低后续实现理解偏差。

- **章节 `foreach` 的能力边界需要更严格约束**  
  当前已说明不支持嵌套 `foreach`，但实现层面还有几个潜在坑：
  - 当绑定参数的值为空数组时，章节是直接不渲染、还是渲染为一条「暂无数据」？  
  - `foreach.as` 的变量作用域如何限定？能否在子章节里继续使用（虽然不支持嵌套 `foreach`，但允许内容中引用）？  
  建议在设计中明确「空列表的渲染策略」以及变量作用域规则，以避免前端/渲染层做各自约定。

- **内容节点的复杂场景尚未覆盖**  
  - 当一个章节既需要图表又需要表格（例如「图 + 表」搭配解释），目前需要拆成多个内容节点还是通过 `composite` 的 higher-level 类型处理？  
  - 自然语言转 SQL (`nl2sql`) 的能力边界未说明：是否允许使用模板参数参与 NL 描述？SQL 生成失败时的降级策略是什么？  
  建议在后续版本补充一些「复杂章节」的模式说明（例如：标准的「图 + 表 + 文字解释」组合）。

- **与权限、数据隔离相关的设计缺失**  
  设计中未提及：
  - SQL 执行的权限边界（例如不同角色对同一模板的字段可见性差异）
  - 分租户 / 分组织（总部 vs 分部）维度下，模板和数据源的隔离策略  
  如果这些属于上游「数据访问层」的职责，也可以在本文中简单声明边界，避免后续误解模板可以随意写 SQL 直连生产库。

## 与实现相关的建议

- **建议补充一个「渲染流水线」的高层时序图**  
  当前对模板和内容结构解释得较好，但缺少对「渲染执行流程」的整体视图，例如：
  - 从参数收集完成后 → 如何遍历 `sections`、解析 `foreach`、触发 SQL / `nl2sql`、再到最终文档（如 Markdown / HTML / DOCX）的生成。
  - 出错/降级路径（某个节点 query 失败时是整体失败还是局部替换占位提示）。  
  一个高层时序图或伪代码有助于统一各模块对执行顺序和错误处理的理解。

- **Schema 约束建议进一步细化并文档化差异**  
  当前已经有 `template.schema.json` 与 YAML 示例，但建议：
  - 在当前设计文档中增加一节「与 schema 的差异说明」，罗列哪些字段是「概念层面已设计但暂未在 v1 schema 中落地」。
  - 对经常可能变更的字段（比如 `presentation` 类型、`chart_type` 枚举）提前声明「可扩展点」，避免使用 `string` 自由值带来后期不兼容。

- **对报表导出目标格式做轻量约束**  
  文档里没有说明最终报告输出形态（仅系统内查看？导出为 docx / pdf / html？），而不同目标会直接影响：
  - 表格能力（合并单元格、自动分页等）
  - 图表渲染方式（内嵌图片 vs 前端渲染组件）  
  建议在当前设计里至少列出 1–2 个「主要支持的目标格式」，并注明对模板能力的影响范围。

- **考虑日志与可观测性**  
  报告生成往往会涉及多次 SQL 和 LLM 请求，建议在设计层面预留：
  - 每次生成任务的 trace id / job id
  - 关键节点（参数解析、SQL 执行、模板渲染）的结构化日志  
  这些可以在后续排查数据问题或用户投诉时提供很大帮助。

## 建议优先补充/澄清的三点

1. **在设计文档中明确模板元信息与模板选择策略**（`type`/`scene` 之外的维度，如版本、启停标记、多模板优先级）。  
2. **在参数与内容节点两个层面补充「校验与失败/降级策略」**（参数缺失/非法、SQL 执行失败、`nl2sql` 解析失败时的处理方式）。  
3. **补充一个端到端的渲染/生成流程视图**，帮助后续实现人员统一对模块边界和调用顺序的理解。
